Project overview

About

testing stereoscopic rendering, you need to focus your eyes on a point behind the screen so that either side overlap in the middle of your view

when successful, you should end up with 3 images in your view, the left-image on the left, the overlapping now slightly 3D image in the middle, and the right image on the right

Nausea potential causes

Viewing angle

high chance the nausea is from being able to see the sphere from a narrower/wider viewing angle than the human eyes/brain are comfortable with

implying that this is some form of existential-nausea/confusion in the muscle memory

which explains why it's not immediate and washes over time in the following 10-50 minutes after viewing

likely from the brain learning to use a narrower/wider distance between the eyes during the viewing (like inverting the steering directions on a bike)

it would then take a moment for the brain to be okay with your natural viewing angle afterwards

Frame rate being too low

also high chance having the frame rate locked at less than 60 fps might be the cause of the nausea issue

it's then the left-right-left-right-left-right blinking of the frame updates which is confusing the brain

this would imply it's the fact the eyes are getting updated information at different speeds to what they're used to

Mixing the two for suffering problem complexity

there's a chance it's the fact it's not mirroring the human eye correctly and it's both the asymmetrical updates AND our lazy method of creating perspective

which leads into the idea that a non-stereoscope perspective has a wider field of view than a single eye would have, so modifying the field of view of either side might be necessary

Suggested solutions

Viewing angle - solution 1

would need to calculate the average distance and angle between the human eyes relative to the viewing distance of the user

Viewing angle - solution 2

having sliding scales for the viewing angle/distance so the user is responsible for this

Frame rate - solution 1

potentially merging the two canvases so that it's all one in a single frame update rather than asymmetrical frame updating

Frame rate - solution 2

updates happen over two frames using interlacing, to "blend" the updating so it appears more like it's simultaneous

OPTION A -- using odds/evens row updates, maybe left is odd and right is even, then next frame left is even and right is odd, repeating the pattern so it's more blended (might just increase the nausea moving into overstimulation territory)

OPTION B -- go for a more cellular interlacing, using (column+row)%2to get a checkerboard interlacing instead of row/columns

Mixing the two - solution 1

updates happen over two frames using interlacing, on a merged canvas

OPTION A -- calculating the angles etc, then using row/column updates in a linear fashion, so that it's even/odd column updating flipflop between frames, but the rows of the updating columns are done across both sides in a single frame

OPTION B -- somewhat the same as OPTION A, but using a random ordering for the column/row updates

Mixing the two - solution 2

similar to solution 1, however using a cellular style of interlacing to give the appearance of a more "analog" update

would likely require more frame buffers to store each frame update, then have the final displayed frame being a combination of these frame buffers

8 should be enough, then using update masks to determine which pixel to take from a given frame buffer

OPTION A -- randomly updating pixels using white noise update mask (simple yes/no if the pixel is taken from that buffer)

OPTION B -- pseudo-random updating pixels using an update mask (some form of fractal noise with alpha blending)
